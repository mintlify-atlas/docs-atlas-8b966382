## Overview

Vista adds two custom data components to store information on `ItemStack` instances:

- **LINKED_FEED_COMPONENT** - Stores a UUID linking an item to a broadcast feed
- **CASSETTE_TAPE_COMPONENT** - Stores a cassette tape holder on cassette items

These components use Minecraft's data component system introduced in 1.20.5+ for persistent and network-synchronized item data.

## LINKED_FEED_COMPONENT

The `LINKED_FEED_COMPONENT` stores a UUID that links an item (typically a hollow cassette) to a specific broadcast feed from a view finder.

### Registration

```java
public static final Supplier<DataComponentType<UUID>> LINKED_FEED_COMPONENT = 
    RegHelper.registerDataComponent(res("linked_feed"), () ->
        DataComponentType.<UUID>builder()
            .persistent(UUIDUtil.CODEC)
            .networkSynchronized(UUIDUtil.STREAM_CODEC)
            .build());
```

## CASSETTE_TAPE_COMPONENT

The `CASSETTE_TAPE_COMPONENT` stores a reference to a cassette tape from the cassette tape registry on cassette items.

### Registration

```java
public static final Supplier<DataComponentType<Holder<CassetteTape>>> CASSETTE_TAPE_COMPONENT = 
    RegHelper.registerDataComponent(res("cassette_tape"), () ->
        DataComponentType.<Holder<CassetteTape>>builder()
            .persistent(RegistryFileCodec.create(VistaMod.CASSETTE_TAPE_REGISTRY_KEY, CassetteTape.DIRECT_CODEC))
            .networkSynchronized(ByteBufCodecs.holderRegistry(VistaMod.CASSETTE_TAPE_REGISTRY_KEY))
            .build());
```

**Location:** `VistaMod.java:134-139`

### Type Information

<ParamField path="Component Type" type="DataComponentType<Holder<CassetteTape>>">
  Data component type storing a holder reference to a cassette tape
</ParamField>

<ParamField path="Persistent Codec" type="Codec<Holder<CassetteTape>>">
  Uses `RegistryFileCodec` for serialization. Stores cassette tape references by registry key (e.g., `"vista:nature"`)
</ParamField>

<ParamField path="Network Codec" type="StreamCodec<ByteBuf, Holder<CassetteTape>>">
  Uses `ByteBufCodecs.holderRegistry()` for efficient network synchronization of registry holders
</ParamField>

### Usage

#### Setting a Cassette Tape

```java
ItemStack cassetteStack = new ItemStack(VistaMod.CASSETTE.get());

// Get a cassette tape holder from the registry
Registry<CassetteTape> registry = level.registryAccess()
    .registryOrThrow(VistaMod.CASSETTE_TAPE_REGISTRY_KEY);
Holder<CassetteTape> tapeHolder = registry.getHolder(
    ResourceKey.create(VistaMod.CASSETTE_TAPE_REGISTRY_KEY,
        ResourceLocation.fromNamespaceAndPath("vista", "nature"))
).orElseThrow();

// Set the component
cassetteStack.set(VistaMod.CASSETTE_TAPE_COMPONENT.get(), tapeHolder);
```

### Component Behavior

<ResponseField name="Persistence">
  The cassette tape reference is saved to NBT as a resource location (e.g., `"vista:nature"`). The full cassette data is loaded from the registry when the world loads.
</ResponseField>

<ResponseField name="Network Sync">
  The holder is efficiently synchronized using registry IDs. The full cassette tape data is available on the client through the synced registry.
</ResponseField>

<ResponseField name="Registry Dependency">
  The component requires the cassette tape registry to be loaded. If a referenced cassette tape doesn't exist, the holder will be empty or invalid.
</ResponseField>

<ResponseField name="Stack Identity">
  Cassettes with different tapes are considered different items and won't stack together.
</ResponseField>

## Complete Examples

### Creating a Cassette Item with a Specific Tape

```java
public ItemStack createCassette(Level level, String tapeId) {
    ItemStack stack = new ItemStack(VistaMod.CASSETTE.get());
    
    Registry<CassetteTape> registry = level.registryAccess()
        .registryOrThrow(VistaMod.CASSETTE_TAPE_REGISTRY_KEY);
    
    Optional<Holder.Reference<CassetteTape>> holder = registry.getHolder(
        ResourceKey.create(VistaMod.CASSETTE_TAPE_REGISTRY_KEY,
            ResourceLocation.parse(tapeId))
    );
    
    holder.ifPresent(h -> stack.set(VistaMod.CASSETTE_TAPE_COMPONENT.get(), h));
    
    return stack;
}
```

## Codec Details

### RegistryFileCodec

The `CASSETTE_TAPE_COMPONENT` uses `RegistryFileCodec.create()` which:

- Serializes holders as resource locations (e.g., `"vista:nature"`)
- Resolves the location to the actual registry entry on deserialization
- Works with both direct values and references
- Supports tags through the holder system

### ByteBufCodecs.holderRegistry()

For network synchronization:

- Uses registry IDs for compact transmission
- Automatically resolves IDs to holders on the receiving end
- Requires both client and server to have synchronized registries
- Handles both direct and reference holders

## Best Practices

<ResponseField name="Null Checks">
  Always check if a component is present before accessing it. `stack.get()` returns `null` if the component is not set.
</ResponseField>

<ResponseField name="Registry Access">
  Obtain registries from `Level.registryAccess()` or `RegistryAccess` instances. Never cache registry instances across world reloads.
</ResponseField>

<ResponseField name="Holder Usage">
  Use `Holder<CassetteTape>` instead of direct `CassetteTape` instances to maintain registry references and tag support.
</ResponseField>

<ResponseField name="Component Removal">
  Use `stack.remove(component)` to remove a component. Setting it to `null` may not work correctly.
</ResponseField>
